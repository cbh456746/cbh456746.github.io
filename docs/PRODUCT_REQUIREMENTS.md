# 제품 요구사항과 변경 불가 원칙

> 상태: Phase 0 기준 문서  
> 적용 범위: 이후의 정보 구조, 디자인, 콘텐츠 이관, 자동화 작업

## 1. 제품 정의

이 사이트는 블로그 하나가 아니라 개인의 장기 플랫폼이다. 다음 네 영역은 서로 다른 목적과 화면을 갖지만 하나의 탐색 체계 안에서 연결된다.

| 영역 | 목적 | 대표 방문자 |
|---|---|---|
| Resume | 짧고 전문적인 자기소개, 현재 집중 분야, 대표 이력 | 교수, 채용 담당자, 협업자 |
| Projects | 결과물, 기술적 선택, 자료, 변경 이력 보관 | 협업자, 프로젝트 사용자 |
| Research / Notes | 수학·통계·AI·데이터 관련 기록 축적 | 학습자, 미래의 자신 |
| Play | 자유로운 사진, 글, 영상, 취미 기록 | 자신과 개인 독자 |

## 2. 홈과 이력의 분리

### 홈 (`/`)

홈은 Soul 페이지다. 개인정보, 학력, 상세 경력, 연락처, 기술 스택을 나열하지 않는다.

- 사진, 짧은 글, 영상, 링크를 자유롭게 배치한다.
- 고정된 카드 수나 섹션 순서를 강요하지 않는다.
- 편집 원본은 하나의 Markdown 파일로 둔다.
- 사진과 영상은 문서화된 단순 Markdown/복사형 삽입 문법으로 추가한다.
- 콘텐츠가 없을 때도 빈 프로필 카드나 자동 최근 글 목록을 강제로 출력하지 않는다.

### Resume (`/resume/` 권장)

개인 정보와 전문성은 한 장짜리 Resume 페이지에 모은다.

1. 짧은 자기소개
2. 현재 집중 분야
3. 연구 관심사
4. 대표 프로젝트
5. 학력·경험·기술
6. 공개를 선택한 연락처와 외부 링크

Resume의 데이터는 한 곳에서 관리하고, 홈에서는 이를 자동 복제하지 않는다.

## 3. 콘텐츠와 태그 정책

### 최상위 콘텐츠 유형

- `Projects`: 지속적으로 관리되는 결과물
- `Research`: 연구·실험·기술 문서
- `Notes`: 학습 정리와 짧은 지식 기록
- `Play`: 일상, 여행, 사진, 책, 영상, 취미

### 태그 규칙

- `Blog`는 사용하지 않는다. 기존 Blog 성격의 글은 `Play`로 이관한다.
- 게임 관련 게시물은 반드시 `Game` 태그를 포함한다.
- 게임 플레이·후기: `Play` + `Game`
- 게임 데이터 분석·개발 프로젝트: `Projects` + `Game` + 관련 기술 태그
- 태그 표기는 제목 형식으로 통일한다. 예: `Play`, `Game`, `Data-Science`, `Web-Scraping`.
- 동일 의미의 복수 표기(`data science`, `Data Science`, `datascience`)는 만들지 않는다.

## 4. 태그 인덱스 요구사항

### PC 우선 배치

글 목록, 아카이브, 태그별 목록에서 좌측에 태그 인덱스 탭을 둔다. 홈은 자유로운 Soul 페이지이므로 이 인덱스를 강제하지 않는다.

```text
좌측: All / Projects / Research / Notes / Play / Game / 세부 태그
우측 상단: 선택된 태그 이름과 설명
우측 본문: 해당 태그 게시물의 제목·날짜·요약·보조 태그 목록
```

- 탭을 누르면 선택 태그의 제목 목록이 콘텐츠 영역 상단에 나타난다.
- 선택 상태는 URL에 남겨 새로고침, 공유, 뒤로 가기에서도 유지한다.
- 각 항목은 제목 전체가 링크여야 하며 키보드만으로 탐색할 수 있어야 한다.
- 태그 표시 순서와 설명은 `_data/tag-index.yml`에서 관리한다.
- 개별 게시물 하단에는 같은 태그의 관련 글을 소수 표시한다.

### 반응형 동작

| 화면 | 인덱스 동작 |
|---|---|
| 1200px 이상 | 좌측 sticky 인덱스 |
| 768–1199px | 콘텐츠 상단의 접이식 인덱스 |
| 767px 이하 | 가로 스크롤 탭 또는 접근 가능한 접이식 패널 |

## 5. 브라우저와 접근성

- 최신 Microsoft Edge를 최우선 검증 브라우저로 한다.
- Chrome, 모바일 Safari에서도 콘텐츠 읽기, 메뉴, 태그 탐색, 검색, 수식 렌더링이 가능해야 한다.
- Internet Explorer 지원은 범위에서 제외한다.
- 키보드 초점, 대체 텍스트, 명시적 레이블, 충분한 색 대비, 모션 감소 설정을 기본으로 한다.

## 6. 유지보수 경험

일반적인 콘텐츠 추가에는 HTML, CSS, Liquid, Ruby, Node.js 지식이 필요하지 않아야 한다.

- 글 유형별 Markdown 템플릿을 제공한다.
- 필수 프런트매터와 잘못된 파일명을 자동 검사한다.
- 이미지는 `assets/images/<slug>/`, 자료는 `assets/files/<slug>/` 규칙을 사용한다.
- 변경은 GitHub Desktop 또는 GitHub 웹 편집기로 수행할 수 있다.
- 자동 검사는 사람이 이해할 수 있는 오류 문구를 제공한다.
- 저장소 문서는 한국어를 기본으로 하고, 코드 식별자와 기술 용어는 필요한 경우 영어를 병기한다.

## 7. 디자인 원칙

- PC에서 넓은 화면과 좌측 인덱스를 활용하되, 본문 줄 길이는 읽기 좋게 제한한다.
- 홈은 자유롭고 감성적이며, Resume/Projects/Research는 절제되고 전문적으로 보이게 한다.
- 대형 배경 이미지는 콘텐츠를 방해하지 않는 경우에만 사용하며, 용도별 크기 제한을 둔다.
- 색상, 여백, 제목, 코드 블록, 표, 인용문은 공통 디자인 토큰으로 관리한다.
- 다크 모드는 시스템 설정을 기본값으로 하되 사용자 선택을 저장할 수 있게 한다.
