## 문제

기존 프로젝트는 마우스 위치에 따라 캐릭터가 25방향으로 움직이고, 마이크 음량에 따라 입 모양이 바뀌는 브라우저 아바타였다. 일반 브라우저에서 사용하는 데는 문제가 없었지만, PRISM Live Studio의 브라우저 소스로 사용하면 실제 마우스 입력이 전달되지 않아 캐릭터 방향이 한곳에 고정됐다.

방송에서 사용하려면 마우스 대신 얼굴 움직임을 지속적으로 전달할 입력 경로가 필요했다. 동시에 실제 눈 추적과 기존 자동 눈깜빡임, 실제 입 추적과 음량 기반 입 모양처럼 여러 입력이 같은 상태를 바꾸는 충돌도 정리해야 했다.

이 작업의 목표는 다음과 같았다.

- iPhone 얼굴 추적으로 캐릭터의 상하·좌우 방향을 바꾼다.
- 실제 눈과 입 움직임을 6개 표정 시트에 연결한다.
- 방송 프로그램의 브라우저 소스에서 투명 배경으로 출력한다.
- VTube Studio가 중단되거나 토큰이 무효화돼도 페이지가 멈추지 않게 한다.
- 공개 저장소로 옮길 수 있도록 개인정보와 작업 파일을 분리한다.

## 접근 방법

프로그램 구조는 [rotejin/tomari-guruguru](https://github.com/rotejin/tomari-guruguru)의 MIT 라이선스 코드를 참고했다. 원본 캐릭터 에셋은 사용하지 않고 직접 권리를 가진 캐릭터 이미지로 교체했다.

캐릭터 이미지는 5행×5열의 25방향과 눈·입 조합 6개로 구성된다. 최종적으로 150개의 WebP 프레임 중 현재 얼굴 상태에 맞는 한 장만 보이도록 전환한다.

### VTube Studio 입력 연결

방송 페이지는 `ws://localhost:8001`의 VTube Studio Public API에 연결한다. 최초 실행 시 플러그인 토큰을 요청하고, 승인받은 토큰은 브라우저의 로컬 저장소에만 보관한다.

50ms 간격으로 다음 다섯 값을 읽는다.

- `FaceAngleX`: 좌우 방향
- `FaceAngleY`: 상하 방향
- `EyeOpenLeft`, `EyeOpenRight`: 눈 감김 상태
- `MouthOpen`: 닫힘·중간·열림의 3단계 입 모양

머리 방향은 바로 프레임을 바꾸지 않고 렌더링 루프에서 보간해 움직임이 급격하게 튀지 않도록 했다.

### 입력 충돌 제거

방송 모드에서는 VTube Studio를 머리·눈·입의 유일한 입력으로 정했다. 기존 자동 눈깜빡임과 음량 기반 입 모양 코드는 토크 모드에만 남기고 방송 모드에서는 제거했다.

추적이 사라졌을 때 임의 애니메이션으로 대체하지 않고 마지막 프레임에서 멈추도록 했다. 실제 얼굴 추적 기반 아바타가 신호를 잃었을 때 움직임이 멈추는 쪽이 더 자연스럽고, 입력 충돌도 완전히 피할 수 있기 때문이다.

### 연결 복구

성공 경로만 있던 초기 연결 코드를 다음 상태로 구분했다.

```text
연결 안 됨 → 연결 중 → 인증 중 → 추적 중 → 재연결 중
```

저장된 토큰이 거부되면 토큰을 삭제하고 새 인증을 요청한다. 연결과 인증에는 시간제한을 두고, 재연결 간격은 1초에서 시작해 최대 30초까지 늘어나도록 했다. VTube Studio가 다시 실행되면 페이지를 새로고침하지 않아도 추적을 재개한다.

## 결과

실제 방송 환경에서 다음 동작을 확인했다.

- iPhone 머리 움직임에 따른 25방향 전환
- 실제 눈 깜빡임과 입 벌림 반영
- PRISM 브라우저 소스의 투명 배경 출력
- VTube Studio 연결 중단 시 마지막 프레임 유지
- 토크 화면에서 방송 화면으로 이동
- 방송 페이지를 포함한 Vite 프로덕션 빌드

공개 준비 과정에서는 원본 시트, 업로드 중간물, 의존성 폴더와 빌드 산출물을 제외한 별도 복사본을 만들었다. 비밀값과 개인 절대경로를 검사하는 스크립트, 150개 프레임과 필수 파일을 확인하는 자가진단도 추가했다.

로컬 루트 경로와 GitHub Pages의 저장소 하위 경로를 각각 가정한 빌드가 모두 통과했다. npm 의존성 감사에서는 확인 시점에 알려진 취약점이 발견되지 않았다.

## 한계와 다음 단계

150개 WebP 프레임의 합계가 약 79MB라 첫 로딩과 저장소 용량이 크다. 프레임을 즉시 전환하기 위해 모두 미리 불러오는 현재 구조는 단순하지만, 장기적으로 이미지 해상도와 압축률을 다시 조정할 필요가 있다.

방송 모드는 로컬 PC의 VTube Studio WebSocket에 의존한다. 따라서 GitHub Pages는 일반 모드의 정적 데모와 빌드 확인에 적합하고, 실제 방송은 로컬 서버로 실행하는 편이 안정적이다.

공개 후에는 GitHub Actions의 깨끗한 Node.js 22 환경 설치 결과, 실제 Pages 주소, 다른 PC의 VTube Studio·PRISM 조합을 추가로 확인할 예정이다.
